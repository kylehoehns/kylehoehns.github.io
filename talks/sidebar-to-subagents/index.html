Gotcha: the Maven reflex

  • What bit us: on a Gradle project, the agent kept reaching for mvn clean install.
  • The fix: an AGENTS.md with project context.
  • The twist: the Claude agent didn't read AGENTS.md natively, so we added a one-line CLAUDE.md that imports it.

What we learned running a team of agents

  • Agentssub-agentssub-agentssub-agents… Delegation nests as deep as you dare. We kept ours one level deep on purpose — the limit now is judgment (and token budget), not the tooling.
  • Parallel agents must be read-only or write to separate areas. Reviewers report; the parent applies fixes.
  • Scope reviewers to the diff (git diff HEAD). Unscoped, they flag pre-existing noise.
  • The bottleneck isn't the parallel part. It's the sequential spine: dev, then test, then fix.
Four tight credibility beats. Lead beat: sub-agents CAN nest now (Claude Code v2.1.172+) — delegation recurses, so the real limit is discipline and cost, not capability. Live example of "the standards are still settling." We deliberately kept our fan-out one level deep; the parent skill (shown earlier) does the orchestration.

Sub-agents are context firewalls

  • The real value isn't parallelism or tidy named "agents" — it's isolation. Each sub-agent gets its own context window.
  • It does the noisy work — reading dozens of files, chasing dead ends — and hands back only a short summary.
  • The parent gets the signal, not the noise. Your main thread stays clean.
  • Even nesting is really about context: more windows to keep clutter out of the one that matters.

Long sessions rot as file reads and tool output pile up. Sub-agents push that noise somewhere else.

The single most important mental model from the field: sub-agents are context hygiene, not a productivity multiplier. "The parent gets the signal, not the noise" (Kyle Redelinghuys). Boris Cherny framed even nested sub-agents as "agents kicking off agents as a way to better manage context." This is why the first sub-agent everyone builds is a read-only explorer.

What teams actually hand to sub-agents

Read-only (report back)

  • Explore the codebase — map how a feature works.
  • Research a library or pattern before you commit.
  • Multi-perspective review: correctness, security, quality.
  • Debug: reproduce, read logs, form a hypothesis.

Scoped writers

  • Generate tests — write access limited to test files.
  • Refactors & migrations — fan out across worktrees.
  • Docs & docstrings — route to a cheaper model.

The litmus test: are the subtasks truly independent, and is the output easy to verify? If not, keep it single-threaded.

The same ~7 roles recur across Claude Code, Copilot, Cursor, Gemini CLI. Codebase exploration is the highest-volume use (Claude Code ships a built-in Haiku-powered read-only Explore agent). Best practice is tight tool-scoping by role: reviewers get Read/Grep only; implementers get Write/Edit/Bash.

From writing code to directing a fleet

  • Worktrees give each agent its own directory and branch — that idea is now a product surface.
  • Cursor's Agents Window, Windsurf / Devin's Command Center, Copilot CLI's fleet mode: dashboards for many agents at once.
  • One pattern won: a lead owns context and decisions, workers return summaries, and writes stay single-threaded.
  • Reality check — the tools give you visibility and isolation. Merge conflicts, ordering, and consistency are still yours.
The 2026 product story: fleets became a first-class UI surface (Cursor 2.0 to 3.x Agents Window + /multitask; Windsurf/Devin Command Center Kanban; Copilot CLI fleet mode, GA Feb 2026). Vendor framing is identical: the engineer's job shifts to directing a fleet. But coordination is still human — the Agents Window "does not merge agent output automatically." The orchestrator-worker / hub-and-spoke pattern is the production default.

Multi-agent is a power tool, not a default

Worth the fan-out

  • High-value work you can actually parallelize.
  • Independent subtasks with verifiable output.
  • Second-opinion review from an agent that didn't write the code.

Keep it single-threaded

  • Tightly coupled work needing constant coordination.
  • Exploratory problems you're still thinking through.
  • Anything you can't meaningfully review.
Capstone honesty beat. Supporting stats: a single agent matches or beats multi-agent on ~64% of tasks (Princeton); multi-agent burns ~15x the tokens of a chat (Anthropic); ~40% of multi-agent pilots fail within six months, usually from picking the wrong pattern. METR (2025): experienced devs on familiar code were ~19% slower with AI while feeling ~20% faster. The delegation-gap stat is the through-line of the whole talk — human judgment is permanent, not transitional.
